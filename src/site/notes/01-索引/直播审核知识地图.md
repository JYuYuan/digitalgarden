# 直播审核知识地图

> 这篇笔记用于串起 TPM 证据审计小程序、Android App、直播对接文档和本次日志补齐方案。

## 核心项目

- [[03-小程序项目/tpm-evidence-audit-wxapp\|tpm-evidence-audit-wxapp]]：Taro 3 小程序端，承载任务详情、排队、直播间、录播、日志上报。
- [[04-移动端项目/tpm-evidence-audit-android\|tpm-evidence-audit-android]]：Android 原生端，承载同一套直播陪审契约和原生日志能力。
- [[05-文档项目/liveup-doc\|liveup-doc]]：直播审核系统对接文档站点，保存后端接口与客户端接入说明。

## 直播主链路

1. 任务列表进入待执行详情。
2. 前置拍照后开始排队。
3. TIM 下发 `pushUrl`，进入直播就绪。
4. 详情页做连通性校验和 TRTC 签名。
5. 进入直播间，完成 TRTC 入房、混流、IM 交互、定位回传。
6. 结束后回到列表，或转录播。

## 跨端共享契约

### 接口

- 任务列表/详情：`live/app/v1/task/list/query`、`live/app/v1/task/detail/query`
- 拍照：`live/app/v1/scene/{taskCode}`、`live/app/v1/scene/execute`、`live/app/v1/scene/query`、`live/app/v1/file/upload`
- 排队：`live/app/v1/line/up`、`live/app/v1/line/cancel`、`live/app/v1/task/currentLineUpMember/query`
- 直播前校验和签名：`live/app/v1/connected/check/{taskCode}`、`live/app/v1/im/sig/create`
- 混流：`live/web/v1/task/mix/start`、`live/web/v1/task/mix/stop`
- 录播：`live/app/v1/video/refuse`、`live/app/v1/video/confirm`、`live/app/v1/video/signature/create`、`live/app/v1/video/record/save`
- 日志：`systemDashboard/v1/log/save`

### TIM key

- 接收：`pushUrl`、`liveStop`、`checkForm`、`msg`、`queueNumber`、`liveQueueStatus`、`fail`、`liveLine`、`grantVideoStatus`、`taskChange`、`getPushUrl`、`getLocation`、`resetLogin`
- 发送：`msg`、`minxStart`、`playUrl`、`checkForm`、`location`、`windowSize`、`locationDistance`

### TRTC 约定

- SDK App ID：`1400380294`
- 用户 ID：`<当前 appUserCode>_auditUserCode<审核员 appUserCode>`
- 房间 ID：`taskCode`
- 流 ID：`taskCode`
- 混流 ID：`mix_<taskCode>_<审核员 appUserCode>`
- 常用混流参数：`videoFramerate: 15`、`videoBitrate: 550`

## 小程序与 Android 差异

- 小程序通过 `trtc-wx-sdk` 生成微信原生 `LivePusher` 参数；Android 使用腾讯原生 `TRTCCloud` 和 `TXCloudVideoView`。
- 小程序 TIM 分发通过 `Taro.getApp()[key]` 加 `setTIMListener(route, code)` 约束页面和任务；Android 由 `IMMessageDispatcher` 按 key 全局分发，再由 Fragment 用 `code == taskCode` 过滤。
- 小程序直播上下文依赖 `Taro.getApp().liveJson`、`photoGPS`、`reportLocation`、`trtcConfig`；Android 使用 `TaskInfo.pushMessage`、`TrtcConfig`、`LocationHelper` 和 `currentTask`。
- Android 额外通过 `TRTCNetworkManager` 监听网络质量并调整码率；小程序主要依赖 TRTC-WX/LivePusher 事件回调。

## 本次日志迭代落点

- 先做统一日志口径，再拆小程序和 Android 的实现。
- P0 优先：崩溃日志、网络日志、拍照执行日志、输入日志。
- P1 扩展：直播日志、录制日志、录播验证、看板对齐。
- 重点要求：所有日志围绕任务号，直播场景额外带会话 ID。

### 日志字段底线

- 必带：任务号、页面、业务场景、事件名称、事件结果、发生时间、端来源。
- Android 额外带：App 版本、设备型号、Android 版本、网络类型。
- 直播额外带：直播会话 ID、房间号、混流任务 ID、IM 消息元信息。
- 禁止记录：Token、密码、图片/视频内容、完整聊天内容、完整地址。

## 改动检查清单

- 改接口、任务状态、TIM key 或 TRTC 参数时，同时检查小程序和 Android。
- 改 `pushUrl`、`getLocation`、`getPushUrl`、`checkForm` 或混流 payload 时，先确认两端 JSON 字段一致。
- 修复任务号误消费、跨页面消息串扰或直播重入时，小程序查 `src/utils/tim.ts` 和页面 `setTIMListener`，Android 查 `IMMessageDispatcher` 和各 Fragment 的 `isInvalidMessage`。
- 调整直播画面、音频、混流或断网策略时，小程序查 `src/components/trtc-room/index.tsx` 和 `liveRoom.tsx`，Android 查 `VideoCallCallingFragment` 和 `TRTCNetworkManager`。
- Android 发版前检查版本号递增、构建产物和发布后台版本一致；敏感账号信息只保留在原发布流程笔记，不在本知识地图重复。
